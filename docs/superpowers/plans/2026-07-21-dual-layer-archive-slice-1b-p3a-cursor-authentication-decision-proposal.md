# Slice 1B-P3A Cursor Authentication Decision Proposal

**Status:** approved, implemented, verified, and formally accepted as part of P3A
**Scope:** P3A R2 cursor authenticity only
**Owner decision:** Decision H1 approved as written
**Runtime effect:** none; the store remains test-constructed and unwired

## Confirmed defect

The current continuation document is bounded and structurally validated but not authenticated. `GHCA_ACD_Archive_Orphan_Reconciler` reads `older_than_epoch` from the caller-supplied cursor and passes it to the store; the store compares the cursor value with that same caller-supplied value. The same problem applies to `after`, `area_index`, traversal positions, and buffered candidates. A valid-looking edit can therefore change eligibility, skip traversal state, or replace the selected frontier without producing `orphan_cursor_invalid`.

This conflicts with approved R2 requirements that the cursor be untrusted and bound to its exact first-call cutoff and private root. Structural validation and candidate revalidation cannot prove that continuation state was issued by this store.

## Decision H1: authenticate the complete canonical cursor

### Recommended contract

Use PHP's standard-library HMAC-SHA-256 implementation in `GHCA_ACD_Private_Archive_Artifact_Store`. Do not add a dependency or persist scan state.

1. Inject one dedicated cursor-authentication key into the private-store constructor. The injected representation is exactly 64 lowercase hexadecimal characters encoding 32 key bytes generated by a CSPRNG.
2. P3A does not load that key from WordPress salts, constants, environment variables, the database, or a file. Production key provisioning and construction remain runtime-wiring work and are not authorized here. P3A tests inject a fixed test-only key.
3. The authenticated payload is the complete existing cursor state, excluding only its authenticator: `cursor_schema_version`, `root_digest`, `older_than_epoch`, `after`, `area_index`, `stack`, and `selected`.
4. The HMAC input is the exact byte sequence `ghca-orphan-cursor-hmac-v1\n` followed by the `ghca-cjson-1` canonical encoding of that payload.
5. The cursor adds one field, `cursor_hmac`, containing the 64-character lowercase hexadecimal result of:

   ```php
   hash_hmac( 'sha256', "ghca-orphan-cursor-hmac-v1\n" . $canonical_payload, $raw_32_byte_key )
   ```

6. Cursor size remains capped at 32,768 canonical bytes including `cursor_hmac`. Existing depth, position, and retained-candidate bounds remain unchanged.
7. On continuation, perform only bounded envelope/canonicalization checks needed to locate and validate `cursor_hmac`; then verify it with `hash_equals()` before any cursor value controls cutoff selection, filtering, traversal, filesystem access, or database recheck.
8. After successful authentication, apply all existing structural, root, cutoff, directory, candidate, containment, and stale-object checks. Authentication supplements those checks; it does not replace them.
9. `GHCA_ACD_Archive_Orphan_Reconciler` computes the safety cutoff only for `cursor = null`. For a continuation it passes no cutoff. The store authenticates the cursor first and then derives the frozen cutoff from the authenticated payload.
10. Amend the store contract to:

    ```php
    GHCA_ACD_Archive_Artifact_Store::enumerate_candidates(
        ?int $older_than_epoch,
        int $limit = 1000,
        ?array $cursor = null
    ): array;
    ```

    A first call requires a non-negative cutoff and no cursor. A continuation requires `older_than_epoch = null` and an authenticated cursor. Any other combination is invalid.
11. An authenticated cursor remains replayable. P3A guarantees tamper-evident deterministic continuation, not single-use consumption. Replaying the same valid cursor may repeat the same report-only work/result but cannot mutate a file, descriptor, or lifecycle row. Strict single-use/exactly-once consumption would require authoritative server-side state and is not part of H1.

### Stable failure behavior

- New construction failure: `orphan_cursor_key_invalid` when the injected key is absent or not exactly 64 lowercase hexadecimal characters.
- Existing continuation failure: `orphan_cursor_invalid` for missing, malformed, wrong-key, or mismatched HMAC and for any authenticated payload that later fails structural validation.
- Existing live-state failure: `orphan_cursor_stale` only after successful authentication when an active directory or buffered candidate has observably changed.
- Authentication failures expose no field-specific mismatch and perform no traversal, artifact mutation, descriptor query, or lifecycle mutation.

### Key stability and rotation

- The same dedicated key must be injected into every process that may continue a cursor for one site.
- P3A supports one active key only. It adds no key ring, key identifier, fallback key, grace window, or persisted key history.
- Rotating the key invalidates in-flight cursors. The caller restarts from `cursor = null`; no retained artifact or database row is affected.
- The key is never included in a cursor, persisted, logged, returned in an exception, or reused as an artifact-encryption/checkpoint-signing key.
- H1 adds no cursor expiry. The signed first-call cutoff remains frozen until completion, restart, or key rotation.

### Reason

HMAC is the minimum root fix: one store boundary issues and verifies every continuation, the PHP 8.3 minimum provides `hash_hmac()` and `hash_equals()`, and the approved 32 KiB bound keeps authentication work bounded. A plain checksum remains forgeable. Server-side scan state would require forbidden schema, cleanup, recovery, and retention contracts.

The authoritative technical design already places HMAC/signing keys outside the database trust boundary and prohibits signing keys in event/task/result payloads. This proposal follows that boundary while deliberately leaving production secret loading unwired.

### Affected code

- `GHCA_ACD_Archive_Artifact_Store::enumerate_candidates()` nullable first-call cutoff contract
- `GHCA_ACD_Private_Archive_Artifact_Store` constructor, cursor issuance, and authentication
- `GHCA_ACD_Archive_Orphan_Reconciler::reconcile()` first-call-only clock/cutoff handling
- P3 storage/orphan tests and traceability only

No schema, plugin entrypoint, hook, scheduler, controller, production handler, runtime configuration loader, network path, deletion, quarantine, or current-site access is authorized.

## Required regression evidence

Each tampering test starts from a genuinely issued cursor, makes a structurally valid edit, and must receive the exact `GHCA_ACD_Archive_Artifact_Store_Exception` with `orphan_cursor_invalid`, zero candidate emission, zero descriptor queries, unchanged database fingerprint, and unchanged filesystem fingerprint:

- `ORPHAN-CURSOR-HMAC-CUTOFF-TAMPER-REJECTED`
- `ORPHAN-CURSOR-HMAC-AFTER-TAMPER-REJECTED`
- `ORPHAN-CURSOR-HMAC-AREA-TAMPER-REJECTED`
- `ORPHAN-CURSOR-HMAC-POSITION-TAMPER-REJECTED`
- `ORPHAN-CURSOR-HMAC-SELECTED-TAMPER-REJECTED`

Also require:

- `ORPHAN-CURSOR-HMAC-VALID-CONTINUATION` — a second store instance with the same injected key continues successfully and preserves the frozen cutoff/order.
- `ORPHAN-CURSOR-HMAC-WRONG-KEY-REJECTED` — a different injected key returns only `orphan_cursor_invalid`.
- `ORPHAN-CURSOR-HMAC-NOT-EXPOSED` — neither the raw key nor its hexadecimal representation appears in cursor canonical JSON, results, or exception text.
- `ORPHAN-CURSOR-HMAC-ROTATION-RESTART` — an old cursor fails after key replacement and a new null-cursor scan succeeds.
- An independent fixed HMAC vector that does not call the production cursor-signing helper.

All existing R2 progress, ordering, size, memory, cutoff, stale-candidate, report-only, and cleanup-boundary tests remain required.

## Retained-data and operational impact

This is an operational cursor-contract change only. P3A cursors are not persisted by the plugin, and no retained event, snapshot, task, artifact descriptor, ledger row, or artifact byte changes. Cursor schema version 1 may be retained because unauthenticated cursors have never been activated in production; every newly issued v1 cursor must include a valid HMAC after this fix.

The existing availability, RPO, RTO, tenancy, data-sensitivity, and lifecycle-command targets remain unchanged. HMAC verification is bounded to one canonical document of at most 32,768 bytes and remains outside synchronous lifecycle-command work.

## Rejected alternatives

- Unkeyed SHA-256 or a root-derived checksum: caller-recomputable and not authentication.
- WordPress `AUTH_SALT`/`NONCE_SALT`: introduces unapproved runtime coupling and rotation behavior.
- A hardcoded plugin key: shared/public and therefore forgeable.
- Persisted database scan state: requires a schema and lifecycle beyond P3A.
- Strict single-use cursor consumption: also requires persisted authoritative state; HMAC alone detects modification but not replay.
- Encrypting cursor contents: confidentiality is unnecessary; the cursor contains immutable IDs and traversal metadata, while authenticity is the missing property.
- Multiple active keys or `key_id`: unnecessary because cursors are ephemeral and restartable.

## Approval and implementation evidence

The repository owner approved **Decision H1 as written** and authorized only the cursor-authentication change, required regressions, available verification matrix, and traceability update. The implementation:

- validates an injected 64-lowercase-hex key and retains only its raw 32 bytes;
- adds a domain-separated HMAC-SHA-256 over all seven canonical cursor payload fields;
- includes the authenticator in the existing 32,768-byte cursor ceiling;
- verifies the authenticator with `hash_equals()` before cutoff, filter, traversal, buffered candidate, filesystem, or descriptor behavior;
- makes the reconciler pass `null` on continuation instead of reading caller-controlled cursor fields;
- keeps rotation restartable, report-only, inactive, and free of runtime key provisioning; and
- preserves the existing schema version because unsigned cursors were never activated or persisted.

All five valid-looking tamper probes, same-key continuation, wrong-key rejection, key rotation/restart, key non-exposure, constructor-key validation, call-shape validation, and independent fixed/issued HMAC vectors pass inside every 840-assertion cell of the definitive Decision C2 matrix: PHP 8.3.30 and PHP 8.5.7 across MySQL 8.0.46, MySQL 8.4.10, and MariaDB 10.6.27. PHP 8.4 remains optional and unavailable because no CLI exists. Independent owner review formally accepted H1 as part of P3A; runtime key provisioning remains unauthorized.
